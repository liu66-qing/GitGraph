# CodeGraph → 代码演化与理解助手 改造方案

> 目标:把现有 CodeGraph(文档知识图谱演化引擎)改造成一个**代码助手**——
> 既能理解代码"现在长什么样"(静态结构),又能追踪代码"怎么演变成现在这样"(git 演化)。
> 灵感来源:GitNexus(浏览器代码知识图谱)。差异化:GitNexus 只有静态快照,我们有**演化 + 冲突检测**。
>
> 适用目标:Agent 应用研发岗位面试作品。

---

## 一、为什么是这个方向

### 1.1 和 GitNexus 的关系:借鉴产品形态,不抄代码

GitNexus 做的事:输入一个代码仓库 → Tree-sitter 解析出代码结构 → 存进浏览器里的 KuzuDB 图数据库 → 内置 Graph RAG Agent 做代码问答。卖点是"零服务器、代码不出浏览器"。

我们**不抄它的实现**(技术栈完全不同:它是 WASM + KuzuDB + Tree-sitter,我们是 FastAPI + Neo4j + Qdrant + LLM),原因有三:

1. 技术栈不兼容,硬搬等于重写;
2. 它是知名开源项目,照搬会被面试官一眼看穿,反成减分项;
3. 静态代码分析是它的主场,我们拼不过,也用不上自己的强项。

我们**借鉴它的产品形态**(代码 → 知识图谱 → Agent 问答),并叠加它结构上做不到的能力:**代码演化**。

### 1.2 我们的独门角度:代码演化(GitNexus 进不来的地)

GitNexus 是一张**静态快照**:代码现在长什么样。它的图谱从当前代码的语法树推导,没有"时间"维度。

我们的项目名字就叫 CodeGraph(演化图谱)。把时序引擎接到 **git 提交历史**上,每个 commit = 一个时间版本,我们就能回答 GitNexus 答不了的问题:

- 这个函数/这个架构是怎么一步步演变成现在这样的?
- 哪次提交引入了破坏性变更(函数签名变了、调用契约破了)?——这正好复用**冲突检测**模块
- 这个模块的依赖关系是从哪几次提交开始膨胀的?

类比:GitNexus 给代码拍**照片**,我们给代码拍**纪录片**。

## 二、现有代码能复用多少(基于真实源码核对)

已逐文件核对 `E:\RAG\src\codegraph` 下的真实接口,复用情况如下。

| 现有模块 | 当前职责 | 代码场景里变成 | 复用程度 |
|---|---|---|---|
| `graph/traversal.py` | 图算法 | `find_path`=调用链追踪, `get_neighborhood`=依赖分析, `get_causal_chain`=影响传播 | **几乎直接复用**,只换语义 |
| `evolution/merger.py` | 时序版本化入图 | 每个 git commit = 一个版本,`valid_from/valid_to/is_active` 直接用 | **结构直接复用** |
| `evolution/conflict_detector.py` | 检测 CEO/收购冲突 | 检测函数签名变更、API 破坏性修改 | **框架复用**,换检测规则 |
| `retrieval/hybrid.py` | 三路混合检索 | 代码混合检索(结构+语义+关键词) | **直接复用** |
| `agent/orchestrator.py` `planner.py` `tools/registry.py` | Agent 编排/规划/工具 | 代码问答 Agent | **直接复用**,加代码专用工具 |
| `evolution/extractor.py` | LLM 抽实体关系 | 降级为"语义层":这个函数/模块是干嘛的 | **部分复用** |
| `evolution/pipeline.py` | 文档处理管线 | 代码处理管线 | **改造**:接入解析器 |
| 前端全部页面 | 图谱/时间线/冲突看板 | 同样的 UI,换数据源 | **直接复用** |
| **需要新写** | —— | **① Python `ast` 代码解析器 ② git 历史读取器** | **新增(约 2 个模块)** |

核心结论:**真正要新写的只有两块——代码结构解析器 + git 历史集成。其余 70% 是改造和复用。**

## 三、一个硬约束,必须先说清楚

`traversal.py` / `merger.py` / `conflict_detector.py` 这些模块的每个函数都在 `await neo4j_client.execute_query(...)`——**它们全部依赖 Neo4j 实际运行**。

要让 demo 真正跑通,环境上绕不开:
- **Neo4j**(图谱存储)
- **Qdrant**(向量检索)
- **Redis**(Celery broker + 缓存)
- **一个 LLM API key**(抽取语义层 + Agent 问答,你的配置是 DashScope)

`docker-compose.yml` 已经定义了前三个。这是"跑通"的最低环境成本,无法省略。如果连这个环境都不想搭,那"跑通 demo"本身不成立——这点我必须诚实告诉你。

## 四、git 历史素材问题(影响 demo 说服力)

实测你当前仓库的 git 历史:

```
2026-05-16  00:35 - 15:57   11 个 commit(CodeGraph 主体,集中在一天)
2026-05-28  17:58 - 18:52   3 个 commit(升级 PulseGraph)
```

问题:这些 commit 大多是 `docs:` / `fix:` / `deploy:`,**真正的功能演化只有少数几个**。直接拿它当"代码演化"demo,讲不出"一个函数怎么一步步演变"的好故事。

三个解法(方案里推荐第 1 个):
1. **拿一个真实的、历史丰富的开源项目当 demo 输入**(比如 fastapi、requests 这类提交历史长的库)。你的引擎是通用的,演示效果立刻拉满,且更有说服力(能处理别人的真实大仓库)。
2. 拿你自己的仓库,但聚焦少数几个有真实改动的 commit 讲。
3. 为了演示,刻意在一个小模块上制造几个有意义的演化 commit。

## 五、分阶段建设路线(关键:每阶段都能跑能演)

混合方向工程量最大。如果三块齐头并进,极可能又卡在"做一半跑不通"——这正是 CodeGraph 现在的处境。所以**目标是混合,但分三阶段,每阶段结束你手上都有一个能跑能演的完整东西。** 面试突然来了也不裸奔。

### 阶段 0:先让现有 CodeGraph 真正跑起来(地基)
不写新功能,只确保现有链路通。没有这步,后面全是空中楼阁。
- 用 docker-compose 起 Neo4j / Qdrant / Redis
- 配好 LLM key,跑通"上传文档 → 抽取 → 入图 → 问答"全链路
- 前端 GraphExplorer / Timeline 从硬编码 demo 数据切换到真实 API
- **产出:一个能跑的文档知识图谱 Agent(就是 CodeGraph 本来该有的样子)**

### 阶段 1:静态代码理解(对标 GitNexus 的核心)
- 新写 `ingestion/code_parser.py`:用 Python 内置 `ast` 解析 .py 文件,提取
  - 节点:Module / Class / Function
  - 边:IMPORTS / CALLS / DEFINES / INHERITS
- 改造 `pipeline.py`:输入从"文档"变成"代码目录",走解析器而非 LLM 抽取(确定性,快且准)
- 复用 `merger.py` 把代码结构入图(节点/边换语义,时序字段先留空)
- 复用 `traversal.py`:`find_path`=调用链, `get_neighborhood`=依赖
- Agent 加代码专用工具:`find_callers`(谁调用了它)、`find_dependencies`、`explain_symbol`
- **产出:输入一个 Python 仓库 → 出调用关系图 + 能问"这函数被谁调用"**

### 阶段 2:代码演化(我们的差异化,GitNexus 没有)
- 新写 `ingestion/git_loader.py`:用 `git log` 遍历提交历史,每个 commit 解析一次代码快照
- 复用 `merger.py` 的时序版本化:commit 时间戳 → `valid_from/valid_to`,旧版本 `is_active=false`
- 改造 `conflict_detector.py` 的规则:
  - 函数签名变更(参数增删)→ 破坏性变更冲突
  - 函数被删除但仍有调用方 → 悬空调用冲突
- 复用前端 Timeline:展示某个函数/模块的演化时间线
- Agent 加工具:`get_symbol_history`(这函数改了几次)、`find_breaking_changes`
- **产出:能问"这个 API 哪次提交被改坏了""这架构怎么演变来的"——纪录片能力**

### 阶段 3:打磨与面试包装
- 语义层:对关键模块用 LLM 生成"这是干嘛的"摘要(复用 extractor)
- 社区检测:把相关代码聚成簇(借鉴 GitNexus + Microsoft GraphRAG)
- 全链路可观测:展示 Agent 的 plan→execute→replan 推理轨迹
- README + 演示脚本 + 面试叙事

**每个阶段都是一个可演示的里程碑。最差情况停在阶段 1,你也有个完整的代码理解助手;走到阶段 2,你就有了别人没有的演化能力。**

## 六、新增的两个核心模块设计

### 6.1 `ingestion/code_parser.py`(阶段 1)
```python
import ast

def parse_python_file(path: str, source: str) -> ParseResult:
    tree = ast.parse(source)
    # 遍历 AST:
    #   ast.FunctionDef / ast.AsyncFunctionDef → Function 节点
    #   ast.ClassDef → Class 节点(+ bases → INHERITS 边)
    #   ast.Import / ast.ImportFrom → IMPORTS 边
    #   ast.Call → CALLS 边(需在函数体内解析被调名)
    # 返回 entities + relations,复用现有 ExtractionResult 结构
```
要点:`ast` 是 Python 内置,零额外依赖,确定性解析(不靠 LLM),又快又准。这本身就是一个面试技术点:"对能规则化的结构用确定性解析,只对语义描述用 LLM"。

## 七、面试怎么讲这个项目

一句话定位:**"一个把代码库变成可演化知识图谱的 Agent——不仅理解代码当前结构,还能追踪它的演化历史,自动发现破坏性变更。"**

可讲的技术深度点(面试官会追问的):
1. **确定性解析 vs LLM 抽取的取舍**:结构用 `ast`(准、快),语义用 LLM(灵活)。体现工程判断力。
2. **时序知识图谱建模**:`valid_from/valid_to/is_active` 三件套实现版本化,commit 映射为时间点。
3. **图算法解决真实问题**:最短路径=调用链,邻居=依赖分析,不是为了用图而用图。
4. **Agent 工具设计**:把检索/分析能力封装成 function calling 工具,Agent 自主规划调用。
5. **冲突检测**:从图结构上自动识别破坏性变更,这是静态分析工具做不到的。

诚实提醒:git 历史素材问题(第四节)若不解决,"演化"这个最大卖点演示效果会打折。建议用一个真实开源项目当 demo 输入。
